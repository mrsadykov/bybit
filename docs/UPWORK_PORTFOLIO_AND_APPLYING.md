# Портфолио и отклики на Upwork

## 1. Какие скрины текущего проекта (торговый бот) лучше в портфолио

Для проекта Bybit/OKX торгового бота логично показать:

| Что снять | Зачем |
|-----------|--------|
| **Dashboard (главная)** | Балансы, метрики, графики PnL — сразу видно «серьёзная панель». Лучше с выбранным периодом (7/30/90 дней) и хотя бы с одной метрикой (боты, сделки, PnL). |
| **Список ботов (Spot / Futures / BTC-quote)** | Показать, что ботов несколько типов, есть статус (активен/неактивен). |
| **Страница редактирования бота** | Параметры (RSI, SL/TP, размер позиции) — видно уровень настройки. |
| **Список сделок (Trades)** | Таблица с датой, парой, стороной, ценой, PnL — доказывает, что система реально торгует и считает. |
| **Логи решений (Decision log)** | Если есть — показывает прозрачность решений (BUY/SELL/SKIP и причина). |

Итого: 4–6 скриншотов. Обязательно: Dashboard + список ботов + сделки. Остальное по желанию. Подписи к скринам на английском: коротко, что на экране (например: "Dashboard – balances, PnL charts, period filter").

## 2. Один проект в портфолио: откликаться сейчас или ждать второй проект?

**Можно откликаться уже с одним сильным проектом.** Важнее качество и ясное описание, чем количество.

- Один детально описанный проект (что сделано, стек, результат) лучше, чем несколько пустых карточек.
- В описании проекта укажи: Laravel, PHP, API биржи (OKX/Bybit), Telegram, дашборд, бэктест, риск-лимиты — это уже сильный кейс для backend/API.

**bala_app:** да, можно добавлять в портфолио по мере готовности: скрины + короткое описание («в разработке», «MVP», «будущее мобильное приложение»). Так видно, что ты не только бота делаешь, а ещё и другой продукт ведёшь. Когда будет версия под App Store — обнови описание и скрины. Не обязательно ждать полного релиза: «In progress» / «MVP» нормально смотрятся, если честно указано.

**Итог:** откликаться можно уже сейчас с одним проектом (торговый бот). Второй (bala_app) добавляй по мере появления экранов и функционала.

## 3. На какие проекты откликаться и на что смотреть

**На что откликаться (под твой профиль Laravel/PHP/API):**

- Backend на Laravel/PHP (API, админки, интеграции).
- Интеграция с внешними API (платёжки, биржи, сервисы).
- Финансовые/торговые системы, боты, дашборды аналитики.
- Задачи «сделать API», «подключить биржу/платёжку», «дашборд с графиками».

**На что обращать внимание в вакансии:**

- Чёткое ТЗ или список задач (не «сделать как в приложении» без деталей).
- Бюджет или hourly rate указаны (хотя бы порядок).
- История клиента: отзывы, сколько нанято, сколько заплатил.
- Язык общения: если только английский — писать на английском.

**Оплата:**

- **Fixed price** — фикс за проект/ milestone. Подходит, когда объём понятен. Деньги через Escrow: заказчик резервирует сумму, после приёмки — тебе.
- **Hourly** — по часам (трекер времени в Upwork). Подходит для длинных или неопределённых задач. Оплата раз в неделю по логам.
- На старте часто предлагают fixed или hourly с нижней ставкой; после отзывов можно поднимать.

**Как договариваться с заказчиком:**

- В отклике: коротко кто ты, почему подходишь под задачу, что уже делал (ссылку на портфолио/проект). Один конкретный вопрос по задаче — показывает интерес.
- Перед началом: зафиксировать объём (что именно делается), сроки, способ оплаты (fixed/hourly), этапы и платежи по milestone при fixed.
- Если fixed — разбить на 1–2 milestone’а с оплатой по результату (например: «API + базовая админка» — первый платёж, «дашборд и отчёты» — второй).
- Общаться в рамках Upwork (чаты, контракты), не переходить на оплату «мимо» платформы — так защищены и ты, и заказчик.

**Кратко:** откликайся на Laravel/API/backend и финансы; смотри на ясность ТЗ, бюджет и репутацию клиента; оплату выбирай fixed или hourly под задачу; всё согласовывай в переписке и через контракт Upwork.
